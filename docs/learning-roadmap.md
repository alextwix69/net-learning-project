# Learning Roadmap

План рассчитан на пошаговое изучение сетей в C++ через небольшой pet-проект. Каждый этап должен заканчиваться рабочим результатом, который можно собрать, запустить и объяснить на собеседовании.

## 1. Подготовка проекта

### 1.1. Минимальная сборка

- Создать первый `src/.../main.cpp` с выводом текста.
- Добавить executable target в `CMakeLists.txt`.
- Собрать проект через:

```bash
cmake -S . -B build
cmake --build build
```

- Запустить получившийся бинарник из `build/`.
- Понять, где лежат исходники, где build artifacts, где executable.

### 1.2. Базовая структура

- Разделить проект на:
  - `include/net/` - заголовочные файлы;
  - `src/common/` - общий код;
  - `src/tcp_echo_client/` - TCP клиент;
  - `src/tcp_echo_server/` - TCP сервер;
  - `src/udp_ping/` - UDP пример;
  - `src/http_client/` - HTTP клиент;
  - `tests/` - тесты.
- Добавить `.gitignore` для `build/`.
- Проверить, что проект можно собрать с нуля после удаления `build/`.

### 1.3. Что нужно уметь объяснить

- Что делает CMake.
- Чем отличается source directory от build directory.
- Что такое target.
- Почему build artifacts не хранят в репозитории.

## 2. Основы Unix sockets

### 2.1. Файловые дескрипторы

- Разобраться, что socket в Linux представлен file descriptor.
- Посмотреть сигнатуры:
  - `socket`;
  - `close`;
  - `send`;
  - `recv`.
- Понять, почему дескриптор нужно закрывать.
- Понять, что будет при утечке file descriptor.

### 2.2. RAII-обертка

- Создать простой класс `Socket`.
- Запретить копирование.
- Разрешить перемещение.
- Закрывать descriptor в деструкторе.
- Добавить методы:
  - `fd()`;
  - `valid()`;
  - `release()`;
  - `reset()`.

### 2.3. Что нужно уметь объяснить

- Что такое RAII.
- Почему socket нельзя просто копировать как `int`.
- Чем отличается ownership от borrowing.
- Почему деструктор удобен для освобождения ресурсов.

## 3. TCP client

### 3.1. Первый TCP connect

- Создать `tcp_echo_client`.
- Принять аргументы командной строки:
  - host;
  - port;
  - message.
- Получить адрес через `getaddrinfo`.
- Создать socket через `socket`.
- Подключиться через `connect`.

### 3.2. Отправка данных

- Отправить строку через `send`.
- Проверить return value.
- Обработать ошибку `send`.
- Добавить функцию `send_all`, которая отправляет все байты.

### 3.3. Чтение ответа

- Прочитать ответ через `recv`.
- Обработать ситуации:
  - получены байты;
  - соединение закрыто;
  - произошла ошибка.
- Вывести ответ в stdout.

### 3.4. Что нужно уметь объяснить

- Что делает `connect`.
- Почему `send` не обязан отправить всю строку за один вызов.
- Почему TCP - это stream, а не message protocol.
- Что означает `recv == 0`.

## 4. TCP echo server

### 4.1. Bind/listen/accept

- Создать `tcp_echo_server`.
- Принять аргументы:
  - bind address;
  - port.
- Создать socket.
- Включить `SO_REUSEADDR`.
- Привязать socket через `bind`.
- Начать слушать через `listen`.
- Принять клиента через `accept`.

### 4.2. Echo logic

- Прочитать данные от клиента.
- Отправить те же данные обратно.
- Закрыть client socket.
- Оставить server socket работать дальше.

### 4.3. Цикл обработки клиентов

- Добавить бесконечный цикл `accept`.
- Обрабатывать одного клиента за раз.
- Проверить поведение с несколькими последовательными клиентами.
- Проверить поведение при пустом сообщении.

### 4.4. Что нужно уметь объяснить

- Чем server socket отличается от client socket.
- Что делает `bind`.
- Что делает `listen`.
- Что возвращает `accept`.
- Почему однопоточный сервер обрабатывает клиентов по очереди.

## 5. Частичные чтение и запись

### 5.1. Partial send

- Реализовать `send_all`.
- Циклом отправлять данные, пока не уйдут все байты.
- Хранить offset отправленных байтов.
- Корректно обрабатывать ошибку.

### 5.2. Partial recv

- Читать данные частями в buffer.
- Собрать полный ответ в `std::string` или `std::vector<char>`.
- Понять, почему один `recv` не равен одному сообщению.

### 5.3. Message framing

- Реализовать один простой способ выделения сообщений:
  - newline-delimited protocol;
  - или length-prefixed protocol.
- Добавить проверку некорректных сообщений.
- Добавить ограничение максимального размера сообщения.

### 5.4. Что нужно уметь объяснить

- Почему TCP не сохраняет границы сообщений.
- Что такое protocol framing.
- Какие плюсы и минусы у newline-delimited protocol.
- Какие плюсы и минусы у length-prefixed protocol.

## 6. UDP пример

### 6.1. UDP sender

- Создать `udp_ping`.
- Принять host, port и message.
- Создать socket с `SOCK_DGRAM`.
- Отправить datagram через `sendto`.

### 6.2. UDP receiver

- Добавить простой UDP server.
- Привязать socket через `bind`.
- Читать datagram через `recvfrom`.
- Выводить адрес отправителя и payload.

### 6.3. Сравнение TCP и UDP

- Проверить, что UDP не требует `connect`.
- Проверить, что UDP сохраняет границу datagram.
- Проверить, что доставка UDP не гарантирована.

### 6.4. Что нужно уметь объяснить

- Чем UDP отличается от TCP.
- Что такое datagram.
- Почему UDP может потерять пакет.
- Где UDP используется на практике.

## 7. Минимальный HTTP client

### 7.1. HTTP request

- Создать `http_client`.
- Подключиться к host:port через TCP.
- Сформировать HTTP/1.1 GET request.
- Обязательно добавить:
  - request line;
  - `Host`;
  - `Connection: close`;
  - пустую строку после headers.

### 7.2. HTTP response

- Читать ответ до закрытия соединения.
- Разделить headers и body по `\r\n\r\n`.
- Распарсить status line:
  - version;
  - status code;
  - reason phrase.
- Распарсить headers в map.

### 7.3. Ограничения первой версии

- Не поддерживать HTTPS.
- Не поддерживать chunked transfer encoding.
- Не поддерживать redirect.
- Явно описать эти ограничения в README.

### 7.4. Что нужно уметь объяснить

- Почему HTTP работает поверх TCP.
- Что такое request line.
- Что такое status line.
- Зачем нужен header `Host`.
- Чем HTTP отличается от raw TCP payload.

### 9.2. CLI и валидация

### 9.3. Логирование

- Логировать:

  - закрытие подключения;

### 9.4. Что нужно уметь объяснить

## 10. Concurrency

### 10.1. Thread per client

- Для каждого accepted client создать отдельный thread.
- Передавать client socket в thread через move.
- Закрывать соединение после обработки.
- Ограничить количество одновременных клиентов или описать проблему.

### 10.2. Event loop

- Разобраться с `poll`.
- Перевести server на обработку нескольких clients в одном thread.
- Хранить список client sockets.
- Удалять закрытые соединения.

### 10.3. Non-blocking IO

- Включить non-blocking mode через `fcntl`.
- Обработать `EAGAIN` и `EWOULDBLOCK`.
- Понять отличие readiness от фактического успешного read/write.

### 10.4. Что нужно уметь объяснить

- Чем blocking IO отличается от non-blocking IO.
- Чем thread-per-client отличается от event loop.
- Какие плюсы и минусы у `poll`.
- Почему `epoll` лучше масштабируется на Linux.
